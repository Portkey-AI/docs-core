[Claude Desktop](https://claude.ai/download) can be configured to use a **Gateway (Anthropic-compatible)** endpoint for third-party inference. Pointing Claude Desktop to Portkey gives you enterprise-grade control for desktop AI usage across teams.

## Why use Portkey with Claude Desktop

- **One gateway, many providers** — Route Claude Desktop traffic to Anthropic, Bedrock, Vertex, and other supported providers through a single endpoint.
- **Centralized governance** — Enforce [budget limits](/product/model-catalog/budget-limits), [rate limits](/product/model-catalog/rate-limits), and workspace-scoped [API keys](https://app.portkey.ai/api-keys).
- **Reliability in production** — Configure [fallbacks](/product/ai-gateway/fallbacks), [load balancing](/product/ai-gateway/load-balancing), [retries](/product/ai-gateway/automatic-retries), and [timeouts](/product/ai-gateway/request-timeouts) without changing end-user apps.
- **Full observability** — Inspect request/response traces, token usage, latency, and spend in [Logs](https://app.portkey.ai/logs) and Analytics.

<Note>
  This guide focuses on Claude Desktop's **Gateway** mode (Anthropic-compatible API). If you are configuring **Claude Cowork** in third-party inference mode, see the dedicated [Claude Cowork integration](/integrations/libraries/claude-cowork).
</Note>

## Architecture (recommended)

`Claude Desktop` → `Portkey Gateway (https://api.portkey.ai)` → `Configured provider(s)`

Recommended pattern:

1. Create provider integrations in Portkey (Anthropic/Bedrock/Vertex/etc.).
2. Create a Portkey Config that sets your default model + reliability policies.
3. Create one or more Portkey API keys attached to that config.
4. Use the API key in Claude Desktop Gateway settings.

This keeps model and policy changes in Portkey, not on each desktop.

## 1) Configure Portkey

<Steps>
  <Step title="Add provider integration(s)">
    In [Model Catalog](https://app.portkey.ai/model-catalog), click **Add Provider** and connect your target provider(s).

    Use clear slugs such as:

    - `anthropic-prod`
    - `bedrock-prod`
    - `vertex-prod`
  </Step>

  <Step title="Create a base config">
    In [Configs](https://app.portkey.ai/configs), create a config with a default model route.

    ```json
    {
      "override_params": {
        "model": "@anthropic-prod/claude-sonnet-4-20250514"
      }
    }
    ```

    Optional (Anthropic prompt caching):

    ```json
    {
      "override_params": {
        "model": "@anthropic-prod/claude-sonnet-4-20250514",
        "cache_control": {
          "type": "ephemeral"
        }
      }
    }
    ```
  </Step>

  <Step title="Attach reliability / governance policies">
    Add guardrails and runtime controls to the config as needed:

    - Fallback chains (primary → backup model/provider)
    - Rate limits and budget limits
    - Request timeout and retry strategy

    This is where Portkey becomes your stable control plane for Claude Desktop traffic.
  </Step>

  <Step title="Create API key(s)">
    In [API Keys](https://app.portkey.ai/api-keys), create a key and attach the config.

    For teams, create separate keys per environment/workspace (for example `claude-desktop-dev`, `claude-desktop-prod`) to isolate usage and limits.
  </Step>
</Steps>

## 2) Configure Claude Desktop (local UI)

Use Claude Desktop's **Developer → Third-Party Inference** flow:

1. Open **Claude Desktop**.
2. In the menu bar, go to **Help → Troubleshooting → Enable Developer mode**.
3. Open **Developer → Configure third-party inference**.
4. Select **Gateway (Anthropic-compatible)**.
5. Enter:
   - **Gateway base URL:** `https://api.portkey.ai`
   - **Gateway API key:** `<YOUR_PORTKEY_API_KEY>`
   - **Gateway auth scheme:** `bearer`
6. Click **Apply locally**.
7. Fully quit Claude Desktop and reopen it.
8. On relaunch, continue with the local/Gateway configuration and send a test request.

After restart, verify requests in [Logs](https://app.portkey.ai/logs).

<Note>
  The exact labels can vary by Claude Desktop version, but the required path is still Developer mode + Third-party inference + Gateway credentials.
</Note>

## 3) Optional: managed rollout (MDM / policy)

For org-managed devices, distribute Gateway settings centrally via your device management flow.

Use the same values as local setup:

- Base URL → `https://api.portkey.ai`
- API key → your scoped Portkey key
- Auth scheme → `bearer`

Keep one managed profile per environment/business unit to maintain clean observability and policy boundaries.

## 4) Route by Config instead of per-request headers

Claude Desktop Gateway mode should be treated as a **fixed client configuration**.

For predictable behavior, do routing through Portkey Configs and API key scoping:

- Create one Config per environment/use case (for example `desktop-default`, `desktop-research`, `desktop-prod`).
- Attach a dedicated API key to each Config.
- Assign users/devices the correct key for their environment.

This avoids relying on ad-hoc per-request header overrides and keeps policy + routing fully governed in Portkey.

### Optional: metadata-based usage policies (Open WebUI-style pattern)

If you want policy controls similar to Open WebUI metadata workflows, use **key segmentation + enforced metadata expectations**:

- Issue separate keys per team/use case (for example `desktop-eng`, `desktop-support`, `desktop-research`).
- Map each key to a config with its own model allowlist, limits, and guardrails.
- Enforce required metadata conventions in your org policy layer (for example environment/team identifiers) using [request metadata controls](/product/administration/enforcing-request-metadata).

Because Claude Desktop Gateway is typically configured as a fixed client, the most reliable pattern is still: **metadata policy by key/workspace boundary**, not per-request overrides from the desktop UI.

## 5) Validate end-to-end

Use this checklist after setup:

1. **Request appears in Logs** with expected API key/workspace.
2. **Model route matches config** (`@provider-slug/model-id`).
3. **Limits are enforced** (rate/budget policies visible in dashboard).
4. **Fallback works** (simulate provider issue and verify backup route).
5. **Cost analytics** are attributed correctly to your Claude Desktop key(s).

## 6) Troubleshooting

- **No logs in Portkey**
  - Confirm base URL is exactly `https://api.portkey.ai` (no extra path).
  - Confirm auth scheme is `bearer`.
  - Verify the API key is active and attached to a config.

- **401/403 errors**
  - Regenerate key and retry.
  - Ensure workspace/org permissions allow that key to use the selected provider/config.

- **Wrong model is being used**
  - Check config `override_params.model`.
  - Remove conflicting custom headers (especially `x-portkey-config` / `x-portkey-provider`) and retest.

- **Intermittent failures/latency spikes**
  - Add fallback routes and automatic retries in config.
  - Set explicit request timeouts and monitor p95 latency in Logs.
