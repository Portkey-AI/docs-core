This guide is for platform teams responsible for secure, scalable Claude Desktop rollout.

## What success looks like

- Teams use Claude Desktop with one governed gateway endpoint
- Metadata-driven policies enforce limits by tenant/team/environment
- Admins can audit behavior, costs, and performance in Portkey

## 1) Set up provider integrations

Go to [Integrations](https://app.portkey.ai/integrations) and connect the providers you need.

Common setup guides:

- [Anthropic integration](/integrations/llms/anthropic)
- [Vertex AI integration](/integrations/llms/vertex-ai)
- [AWS Bedrock integration](/integrations/llms/bedrock/aws-bedrock)

## 2) Create configs (why this matters)

Configs are your policy/routing control point. They let you change model behavior centrally without touching every desktop.

Create configs by team/environment (for example `desktop-eng-prod`, `desktop-support-prod`, `desktop-dev`).

Example:

```json
{
  "override_params": {
    "model": "@anthropic-prod/claude-sonnet-4-20250514"
  }
}
```

## 3) Define governance (what and how)

Apply governance at config/workspace level:

- **Budget limits** to cap spend by cohort
- **Rate limits** to protect infrastructure and fair usage
- **Guardrails** for content/security controls

Recommended rollout pattern:

1. Start with moderate limits in staging.
2. Validate user workflows and false positives.
3. Tighten limits/guardrails in production per team risk profile.

## 4) Create and distribute scoped API keys

Create keys per team/environment (example: `claude-desktop-eng-prod`).

Each key should map to:

- the right config,
- the right workspace scope,
- the right budget/rate controls.

Deliver to users with one copy-paste template for Gateway settings.

## 5) Metadata-based usage policy (core lighthouse pattern)

Claude Desktop users set one extra header:

- `x-portkey-metadata`

Example value you can standardize across teams:

```json
{"tenant":"acme","user":"alice@acme.com","team":"support","env":"prod"}
```

<Tabs>
  <Tab title="Why metadata matters">

Metadata is the policy context. It enables:

- Per-team limits (`team=support` vs `team=engineering`)
- Per-environment controls (`env=dev` vs `env=prod`)
- Clear attribution (`user`, `tenant`) for audits and cost analytics

  </Tab>

  <Tab title="Policy examples you can ship">

- `team = support` → allow only approved Claude model + lower RPM
- `env = dev` → cheaper model default + lower budget cap
- `tenant = enterprise-a` → dedicated config + stricter guardrails

  </Tab>

  <Tab title="User rollout template">

Tell users to configure:

- Gateway URL: `https://api.portkey.ai`
- Auth scheme: `bearer`
- API key: team-issued key
- Extra header:

```json
x-portkey-metadata: {"tenant":"acme","user":"<email>","team":"support","env":"prod"}
```

  </Tab>
</Tabs>

## 6) Observe outcomes (fruit of admin work)

After rollout, use Portkey to continuously improve:

- **Logs:** verify routing decisions and policy triggers
- **Analytics:** compare spend, tokens, and latency by team/tenant/env
- **Operations:** tune limits and model routing based on real usage patterns

## 7) Validation checklist

- Requests include expected metadata keys.
- Metadata conditions trigger the intended policy behavior.
- Spend and rate limits behave correctly per team.
- Routing/fallback behavior matches each config.
- Teams can ship without policy bypasses.
