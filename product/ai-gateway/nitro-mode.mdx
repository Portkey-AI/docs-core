The Portkey Gateway supports **Nitro mode** through the `x-portkey-nitro-mode` header. When this header is set, the request body is streamed directly to the provider instead of being opened or transformed by the gateway.

This helps reduce latency and improves throughput. Since the gateway does not read the request body in this mode, some features are not available. See [Limitations](#limitations) for more details.

## How It Works

Normally, the gateway reads the request body so it can apply features like transformations, retries, fallbacks, guardrails, and other body-based configurations.

When `x-portkey-nitro-mode` header is set, the gateway **skips reading the request body entirely** and streams it directly to the provider. This removes parsing and processing overhead, allowing requests to reach the provider faster.

Because the gateway never reads the body in this mode, it cannot apply any features that depend on inspecting or modifying the request payload. That’s why Nitro mode comes with a few limitations.

## When to Use Nitro Mode

Use Nitro mode when you want the **lowest possible latency** and are sending requests to a **single provider**.

If your use case depends on features like **fallbacks, retries, load balancing, request transformations, or input guardrails**, use the standard mode instead.

## Example

Add the header to your request:

| Header | Value |
|--------|--------|
| `x-portkey-nitro-mode` | `true` |

<CodeGroup>

```sh cURL
curl https://api.portkey.ai/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "x-portkey-api-key: $PORTKEY_API_KEY" \
  -H "x-portkey-provider: @openai-prod" \
  -H "x-portkey-nitro-mode: true" \
  -d '{"model": "gpt-4o", "messages": [{"role": "user", "content": "Hello"}]}'
```

```py Python
from portkey_ai import Portkey

portkey = Portkey(
    api_key="PORTKEY_API_KEY",
    provider="@openai-prod"
)

response = portkey.with_options(
    extra_headers={"x-portkey-nitro-mode": "true"}
).chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Hello"}]
)
```

```js JavaScript
import Portkey from 'portkey-ai';

const portkey = new Portkey({
    apiKey: "PORTKEY_API_KEY",
    provider: "@openai-prod"
});

const response = await portkey.chat.completions.create(
    { model: "gpt-4o", messages: [{ role: "user", content: "Hello" }] },
    { headers: { "x-portkey-nitro-mode": "true" } }
);
```

</CodeGroup>

## Supported Endpoints

Nitro mode works only on the following endpoints:

| Endpoint | Description | Currently Supported Providers |
|----------|-------------|---------------------|
| `/v1/chat/completions` | Chat completions | OpenAI, Fireworks, OpenRouter |
| `/v1/completions` | Completions | OpenAI, Fireworks, OpenRouter |
| `/v1/embeddings` | Embeddings | OpenAI, Fireworks, OpenRouter |
| `/v1/images/generations` | Image generation | OpenAI |
| `/v1/audio/speech` | Text-to-speech | OpenAI |
| `/v1/messages` | Messages API | Anthropic |
| `/v1/responses` | Responses API | OpenAI, OpenRouter |

## Limitations

Because the gateway does not read the request body in Nitro mode, any feature that depends on inspecting or modifying the request payload is not available.

The provider must be specified either through the `x-portkey-provider` header (with the provider slug) or through the config's `provider` option. Passing the provider in the request body is not supported in Nitro mode since the body is never read.

Additionally, the following configuration constraints apply. If any of these are violated, the gateway returns an error. Fix the configuration or omit the header to use standard behavior.

- **Single target.** Use exactly one target. Fallbacks, load balancing, conditional strategies, and multiple targets are not supported current.
- **No retries.** Set `retry.attempts` to `0` or omit retries entirely. 
- **No caching.** Caching is not supported, as it requires reading the full request body to build cache keys.
- **No body-dependent hooks.** Hooks that read the request body (such as `input_guardrails` or `before_request_hooks`) are not supported.
  - This also applies to default input guardrails set at the workspace or organisation level.
- **No body override parameters.** `override_params` that change request body fields (such as `model` or `temperature`) are not supported.

## Errors

If your configuration breaks any of these rules, the gateway does not forward the request and returns an HTTP 4xx error with a message describing the issue (for example: single target required, retries not supported, config must be passed via headers, or provider not allowed for that API).

Fix the configuration as indicated in the error message, or remove the `x-portkey-nitro-mode` header to use standard gateway behavior.
