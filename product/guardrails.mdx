Guardrails help you **control what goes into an LLM and what comes out**.

If you’re new to guardrails, the easiest way to think about them is: **a set of checks + a set of actions**.

- **Checks** answer “is this prompt/response okay?” (pass/fail)
- **Actions** answer “what should we do if it fails?” (log, warn, block, retry, fall back, etc.)

Guardrails run **at the Gateway layer** (between your app and the model), so the same Guardrail works across providers. They also combine cleanly with other Gateway orchestration like retries, fallbacks, caching, timeouts, and load balancing.

#### A couple of examples

- **JSON Schema (output guardrail)**: ensure the model returns valid JSON in the shape your app expects.
- **Prompt Injection (input guardrail)**: detect jailbreak / instruction-injection attempts before the prompt hits the model.

Portkey offers 20+ built-in checks (deterministic + LLM-based) and integrations with guardrail partners.

## Using Guardrails

Putting Portkey Guardrails in production is just a 4-step process:

1. Create Guardrail Checks
2. Create Guardrail Actions
3. Enable Guardrail through Configs
4. Attach the Config to a Request

### The 2 concepts to remember

- **Checks**: small validations that return a boolean verdict (pass/fail).
- **Actions**: what Portkey should do when checks pass or fail (log, warn via status codes, deny, collect feedback, etc.).

This flowchart shows how a request moves through Portkey Guardrails:

- Your app sends a request to Portkey.
- If you enabled **input guardrails**, Portkey evaluates the input before forwarding to the LLM.
- Portkey calls the LLM.
- If you enabled **output guardrails**, Portkey evaluates the output before returning it to your app.
- Based on your action settings (async/deny/etc.), Portkey may simply log the result, return a warning status, or block.

<Note>
  For chat requests, Portkey evaluates the **last message** in the request body when running guardrail checks.
</Note>

## 2\. Add Guardrail Actions

Actions define what Portkey should do with the result of your checks.

<Info>
	This Guardrail validates **either** the input **or** the output — so the actions you set here apply to that same direction (request or response).
</Info>

<Frame>
  <img src="/images/product/ai-gateway/ai-33.png"/>
</Frame>

### Guardrail actions (and what they change)

These settings apply to the Guardrail direction you chose:

- **Input guardrail**: runs on the request, before it reaches the LLM.
- **Output guardrail**: runs on the response, before it reaches your app/user.

#### Core action settings

| Action | State | Meaning (in general) | Impact on input guardrails | Impact on output guardrails | Common use case |
|:--|:--:|:--|:--|:--|:--|
| **Async** | `true` (default) | Run the checks in the background (in parallel with the LLM call). The verdict does **not** change what gets returned. | The request still goes to the LLM. You can review results in logs. | The response is returned as-is. You can review results in logs. | “Observe first” rollout: measure how often your traffic fails before you start blocking or retrying. |
| **Async** | `false` | Run checks inline (Portkey waits for the verdict). This lets you **warn** (`246`) or **block** (`446`) on failures. Adds latency. | Checks run before the request is sent upstream. Can block a bad prompt before it hits the model. | Checks run before returning the model response. Can block a bad response before your app sees it. | Safety/compliance gates, or any workflow where you want fallbacks/retries based on guardrail failures. |
| **Deny** | `false` (default) | If checks fail (and `async=false`), Portkey still proceeds but returns **246** so you can react. | The request still proceeds (with status `246` if it fails). | The response still proceeds (with status `246` if it fails). | “Warn and handle”: route to fallback model, retry, degrade experience, alert, etc. |
| **Deny** | `true` | If checks fail (and `async=false`), Portkey blocks the request/response with **446**. | Blocks before sending the request upstream. | Blocks before returning the response to your app. | Hard-stop unsafe or non-compliant traffic (PII leakage, jailbreaks, policy violations). |
| **Sequential** | `false` (default) | Run checks in parallel. Usually fastest. | Faster guardrail evaluation (when checks are independent). | Faster guardrail evaluation (when checks are independent). | Default choice for most Guardrails. |
| **Sequential** | `true` | Run checks one-by-one in order. Usually slower, but predictable. | Useful if one check depends on another or you want a stable order. | Useful if one check depends on another or you want a stable order. | Dependent checks, debugging, or when ordering matters more than latency. |

#### Feedback settings (optional)

| Action | State | Meaning (in general) | Impact on input guardrails | Impact on output guardrails | Common use case |
|:--|:--|:--|:--|:--|:--|
| **On success** | configured / not configured | When all checks pass, Portkey attaches a feedback object to help you track “good” traffic. | Feedback is attached for passed input checks. | Feedback is attached for passed output checks. | Build an eval dataset over time (what “good” looks like). |
| **On failure** | configured / not configured | When any check fails, Portkey attaches a feedback object to help you track “bad” traffic. | Feedback is attached for failed input checks. | Feedback is attached for failed output checks. | Build an eval dataset over time (what fails, how often, and why). |

### Guardrail behaviour on the Gateway (status codes + impact)

This table answers the most common question: **“If I set async / deny / sequential like this, what status code will I see and what actually happens?”**

<Note>
  Status codes below describe guardrail outcomes when the upstream LLM call would otherwise succeed. Provider-side errors (4xx/5xx) are still returned as-is.

  `async` and `deny` are mutually exclusive: when `async=true`, `deny` has no effect (and should be `false`). `sequential` can be `true` or `false` in any mode.
</Note>

| Async | Deny | Sequential | Effect on the LLM call (input guardrails) | Effect on the LLM call (output guardrails) | Status on PASS | Status on FAIL | Blocked on FAIL | Adds request latency | Hook results in API response | Results in Portkey Logs |
|:--:|:--:|:--:|:--|:--|:--|:--|:--:|:--|:--:|:--:|
| true | false | false | Request processed (not affected by guardrail verdict). | Response returned (not affected by guardrail verdict). | Provider status code | Provider status code | No | No | No | Yes |
| true | false | true  | Request processed (not affected by guardrail verdict). | Response returned (not affected by guardrail verdict). | Provider status code | Provider status code | No | No | No | Yes |
| false | false | false | Request processed even if checks fail. | Response returned even if checks fail. | 200 | 246 (**failed, but allowed**) | No | Yes (parallel checks) | Yes | Yes |
| false | false | true  | Request processed even if checks fail. | Response returned even if checks fail. | 200 | 246 (**failed, but allowed**) | No | Yes (sequential checks) | Yes | Yes |
| false | true  | false | Request blocked (LLM not called) if checks fail. | Response blocked if checks fail (LLM already called). | 200 | 446 (**failed, denied**) | Yes | Yes (parallel checks) | Yes | Yes |
| false | true  | true  | Request blocked (LLM not called) if checks fail. | Response blocked if checks fail (LLM already called). | 200 | 446 (**failed, denied**) | Yes | Yes (sequential checks) | Yes | Yes |

Set the actions you want, name your Guardrail, and save. You’ll get a `$GUARDRAIL_ID` that you attach to requests via Configs.

## 3\. "Enable" the Guardrails through Configs

This is where Portkey's magic comes into play. The Guardrail you created above is yet not an `Active` guardrail because it is not attached to any request.

Configs is one of Portkey's most powerful features and is used to define all kinds of request orchestration - everything from caching, retries, fallbacks, timeouts, to load balancing.

<Info>

	Now, you can use Configs to add **Guardrail checks** & **actions** to your request.
</Info>




### Option 1: Direct Guardrail Configuration (Recommended)

Portkey now offers a more intuitive way to add guardrails to your configurations:

| Config Key           | Value                                 | Description                                                      |
|:---------------------|:--------------------------------------|:-----------------------------------------------------------------|
| **input_guardrails** | `["guardrails-id-xxx", "guardrails-id-yyy"]` | Apply these guardrails to the **INPUT** before sending to LLM |
| **output_guardrails**| `["guardrails-id-xxx", "guardrails-id-yyy"]` | Apply these guardrails to the **OUTPUT** from the LLM       |

```json
{
  "retry": {
    "attempts": 3
  },
  "cache": {
    "mode": "simple"
  },
  "provider":"@openai-xxx",
  "input_guardrails": ["guardrails-id-xxx", "guardrails-id-yyy"],
  "output_guardrails": ["guardrails-id-xxx", "guardrails-id-yyy"]
}
```

### Option 2: Hook-Based Configuration (For Creating [Raw Guardrails](/product/guardrails/creating-raw-guardrails-in-json))

You can also continue to use the original hook-based approach:

| Type                | Config Key                | Value                       | Description                                                           |
|:------------------- |:--------------------------|:----------------------------|:----------------------------------------------------------------------|
| Before Request Hook | **before_request_hooks**  | `[{"id":"$guardrail_id"}]`  | These hooks run on the **INPUT** before sending to the LLM            |
| After Request Hook  | **after_request_hooks**   | `[{"id":"$guardrail_id"}]`  | These hooks run on the **OUTPUT** after receiving from the LLM        |


```json
{
  "retry": {
    "attempts": 3
  },
  "cache": {
    "mode": "simple"
  },
  "provider":"@openai-xxx",
  "before_request_hooks": [{
    "id": "input-guardrail-id-xx"
  }],
  "after_request_hooks": [{
    "id": "output-guardrail-id-xx"
  }]
}
```

<Note>
Both configuration approaches work identically - choose whichever is more intuitive for your team. The simplified `input_guardrails` and `output_guardrails` fields are recommended for better readability.
</Note>

### Example Config Using the New `246` & `446` Status Codes


<Tabs>
  <Tab title="Fallback to Another Model on Guardrail Fail">
```json
{
  "strategy": {
    "mode": "fallback",
    "on_status_codes": [246, 446]
  },
  "targets": [
    {"provider":"@openai-key-xxx"},
    {"provider":"@anthropic-key-xxx"}
  ],
  "input_guardrails": ["guardrails-id-xxx"]
}
```
  </Tab>
  <Tab title="Retry on Guardrail Fail (Simplified)">
```json
{
  "retry": {
    "on_status_codes": [246],
    "attempts": 5
  },
  "output_guardrails": ["guardrails-id-xxx"]
}
```
  </Tab>
</Tabs>


Create these Configs in Portkey UI, save them, and get an associated Config ID to attach to your requests. [More here](/product/ai-gateway/configs).

## 4\. Final Step - Attach Config to Request

Now, while instantiating your Portkey client or while sending headers, just pass the Config ID.

<CodeGroup>
```js Node
const portkey = new Portkey({
    apiKey: "PORTKEY_API_KEY",
    config: "pc-***" // Supports a string config id or a config object
});
```

```py Python
portkey = Portkey(
    api_key="PORTKEY_API_KEY",
    config="pc-***" # Supports a string config id or a config object
)
```

```js OpenAI Node
const openai = new OpenAI({
  apiKey: 'OPENAI_API_KEY',
  baseURL: PORTKEY_GATEWAY_URL,
  defaultHeaders: createHeaders({
    apiKey: "PORTKEY_API_KEY",
    config: "CONFIG_ID"
  })
});
```

```py OpenAI Python
client = OpenAI(
    api_key="OPENAI_API_KEY", # defaults to os.environ.get("OPENAI_API_KEY")
    base_url=PORTKEY_GATEWAY_URL,
    default_headers=createHeaders(
        provider="openai",
        api_key="PORTKEY_API_KEY", # defaults to os.environ.get("PORTKEY_API_KEY")
        config="CONFIG_ID"
    )
)
```

```sh cURL
curl https://api.portkey.ai/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "x-portkey-api-key: $PORTKEY_API_KEY" \
  -H "x-portkey-config: $CONFIG_ID" \
  -d '{
    "model": "gpt-3.5-turbo",
    "messages": [{
        "role": "user",
        "content": "Hello!"
      }]
  }'
```
</CodeGroup>
