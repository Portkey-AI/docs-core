---
title: "Creating Raw Guardrails (in JSON)"
description: Define guardrail checks & actions directly in your request/config JSON (no UI required).
---

Raw guardrails are the way to use Portkey Guardrails by implementing them via code.

Instead of creating a Guardrail in the dashboard and referencing a Guardrail ID, you embed the guardrail definition directly inside your Config (or `x-portkey-config` header).

This is useful when:

- You want **infrastructure-as-code** guardrails (reviewed, versioned, deployed with your app).
- You want the **same checks** but **different actions** depending on endpoint, customer tier, or environment.
- Your check parameters are computed in code (for example, a dynamic allowlist/denylist).
- You’re prototyping and want to iterate quickly without touching the UI.

<Note>
  Raw guardrails use the same runtime behavior described in the Guardrails overview (status codes `246`/`446` for synchronous failures). See [Guardrails](/product/guardrails).
</Note>

---

## How raw guardrails map to input vs output

- **Input guardrails** live in `before_request_hooks` (run before the request goes to the LLM).
- **Output guardrails** live in `after_request_hooks` (run after the LLM responds, before Portkey returns to your app).

---

## Example: input guardrail (warn, don’t block)

This example checks the input with a regex and **warns** on failure (`246`) instead of blocking.

```json
{
  "before_request_hooks": [
    {
      "type": "guardrail",
      "id": "my_input_guardrail",
      "checks": [
        {
          "id": "default.regexMatch",
          "parameters": {
            "rule": "\\b(jailbreak|ignore previous instructions)\\b",
            "not": false
          }
        }
      ],
      "async": false,
      "deny": false,
      "sequential": false
    }
  ]
}
```

---

## Example: output guardrail (block on failure)

This example validates the model output and **blocks** if it fails (`446`).

```json
{
  "after_request_hooks": [
    {
      "type": "guardrail",
      "id": "my_output_guardrail",
      "checks": [
        {
          "id": "default.jsonSchema",
          "parameters": {
            "schema": {
              "type": "object",
              "properties": { "answer": { "type": "string" } },
              "required": ["answer"],
              "additionalProperties": false
            }
          }
        }
      ],
      "async": false,
      "deny": true,
      "sequential": false
    }
  ]
}
```

## Understanding the action fields

### `async`

- `async=true`: checks run in the background → request/response is **not affected** (results show in logs).
- `async=false`: checks run inline → Portkey can **warn** (`246`) or **block** (`446`) based on `deny`.

### `deny`

- `deny=true`: if checks fail (and `async=false`) → Portkey **blocks** with `446`.
- `deny=false`: if checks fail (and `async=false`) → Portkey **continues** but returns `246`.

### `sequential`

- `sequential=false` (default): run checks in parallel (faster).
- `sequential=true`: run checks one-by-one (more predictable, slower).

### `on_success` and `on_fail` (feedback)

Use these to attach feedback metadata (useful for eval datasets and analytics):

```json
{
  "before_request_hooks": [
    {
      "type": "guardrail",
      "id": "my_guardrail_with_feedback",
      "checks": [{ "id": "default.regexMatch", "parameters": { "rule": ".*", "not": false } }],
      "async": false,
      "deny": false,
      "sequential": false,
      "on_success": { "feedback": { "value": 1, "weight": 1, "metadata": "" } },
      "on_fail": { "feedback": { "value": -1, "weight": 1, "metadata": "" } }
    }
  ]
}
```
