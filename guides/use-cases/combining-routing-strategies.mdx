title: "Combining Routing Strategies: Conditional, Load Balancing & Fallbacks"
description: "Route by model, distribute across providers, and fail gracefully — all in one Gateway Config."
---

Production AI apps need more than one routing strategy:

- **Route by model** — send `claude-sonnet` requests to Anthropic's ecosystem and `gpt-4o` requests to OpenAI's, from a single API call
- **Load balance within each model** — spread traffic across Anthropic, Vertex AI, and Bedrock to multiply rate limits
- **Fail gracefully** — if an entire provider cluster goes down, fall back to a reliable backup automatically

Portkey's [Gateway Configs](/product/ai-gateway/configs) support nesting all three strategies together.

## How Nesting Works

Each config target can itself contain a strategy. A conditional routing target can point to a load balancer; a load-balanced target can be wrapped in a fallback:

```
Conditional Router (by params.model)
├── "claude-sonnet"  → Load Balancer
│     ├── Anthropic (direct)
│     ├── Vertex AI (Claude on Vertex)
│     └── AWS Bedrock (Claude on Bedrock)
│
├── "gpt-4o"         → OpenAI (direct)
│
├── "gpt-4o-mini"    → Load Balancer
│     ├── Azure OpenAI
│     ├── OpenAI (key 1)
│     └── OpenAI (key 2)
│
└── "gemini-2.5-pro" → Fallback
      ├── Load Balancer (primary)
      │     ├── Vertex AI (Gemini)
      │     ├── Google AI Studio (key 1)
      │     └── Google AI Studio (key 2)
      └── OpenAI gpt-4.1 (if all Gemini endpoints fail)
```

## The Complete Config

Save this in the [Portkey UI](https://app.portkey.ai/configs) and copy the resulting Config ID.

```json
{
  "strategy": {
    "mode": "conditional",
    "conditions": [
      { "query": { "params.model": { "$eq": "claude-sonnet" } }, "then": "claude-sonnet-lb" },
      { "query": { "params.model": { "$eq": "gpt-4o" } }, "then": "gpt-4o-target" },
      { "query": { "params.model": { "$eq": "gpt-4o-mini" } }, "then": "gpt-4o-mini-lb" },
      { "query": { "params.model": { "$eq": "gemini-2.5-pro" } }, "then": "gemini-lb-with-fallback" }
    ],
    "default": "gpt-4o-target"
  },
  "targets": [
    {
      "name": "claude-sonnet-lb",
      "strategy": { "mode": "loadbalance" },
      "targets": [
        { "provider": "@anthropic-vk", "override_params": { "model": "claude-sonnet-4-5-20250514" }, "weight": 1 },
        { "provider": "@vertex-vk",    "override_params": { "model": "claude-sonnet-4-5@20250514" }, "weight": 1 },
        { "provider": "@bedrock-vk",   "override_params": { "model": "anthropic.claude-sonnet-4-5-20250514-v1:0" }, "weight": 1 }
      ]
    },
    {
      "name": "gpt-4o-target",
      "provider": "@openai-vk",
      "override_params": { "model": "gpt-4o" }
    },
    {
      "name": "gpt-4o-mini-lb",
      "strategy": { "mode": "loadbalance" },
      "targets": [
        { "provider": "@azure-vk",    "override_params": { "model": "gpt-4o-mini" }, "weight": 1 },
        { "provider": "@openai-vk-1", "override_params": { "model": "gpt-4o-mini" }, "weight": 1 },
        { "provider": "@openai-vk-2", "override_params": { "model": "gpt-4o-mini" }, "weight": 1 }
      ]
    },
    {
      "name": "gemini-lb-with-fallback",
      "strategy": { "mode": "fallback" },
      "targets": [
        {
          "strategy": { "mode": "loadbalance" },
          "targets": [
            { "provider": "@vertex-vk",   "override_params": { "model": "gemini-2.5-pro" }, "weight": 1 },
            { "provider": "@google-vk-1", "override_params": { "model": "gemini-2.5-pro" }, "weight": 1 },
            { "provider": "@google-vk-2", "override_params": { "model": "gemini-2.5-pro" }, "weight": 1 }
          ]
        },
        { "provider": "@openai-vk", "override_params": { "model": "gpt-4.1" } }
      ]
    }
  ]
}
```

## Breaking Down Each Target

### Target 1: `claude-sonnet-lb` — Load Balanced Across Three Providers

Each provider runs Claude Sonnet 4.5 with independent rate limit buckets. Equal weights (`1:1:1`) split traffic evenly — effectively tripling throughput compared to a single-provider setup.

| Provider | Virtual Key slug | Model identifier |
|----------|-----------------|------------------|
| Anthropic (direct) | `@anthropic-vk` | `claude-sonnet-4-5-20250514` |
| Google Vertex AI | `@vertex-vk` | `claude-sonnet-4-5@20250514` |
| AWS Bedrock | `@bedrock-vk` | `anthropic.claude-sonnet-4-5-20250514-v1:0` |

Each provider uses a slightly different model identifier format — `override_params.model` handles this per target.

```json
{
  "name": "claude-sonnet-lb",
  "strategy": { "mode": "loadbalance" },
  "targets": [
    { "provider": "@anthropic-vk", "override_params": { "model": "claude-sonnet-4-5-20250514" }, "weight": 1 },
    { "provider": "@vertex-vk",    "override_params": { "model": "claude-sonnet-4-5@20250514" }, "weight": 1 },
    { "provider": "@bedrock-vk",   "override_params": { "model": "anthropic.claude-sonnet-4-5-20250514-v1:0" }, "weight": 1 }
  ]
}
```

### Target 2: `gpt-4o-target` — Direct Single Provider

No nested strategy. When `params.model` is `gpt-4o`, the request goes straight to OpenAI. Use for models where a single provider's rate limits aren't a bottleneck.

```json
{
  "name": "gpt-4o-target",
  "provider": "@openai-vk",
  "override_params": { "model": "gpt-4o" }
}
```

### Target 3: `gpt-4o-mini-lb` — Three Accounts for Maximum Rate Limits

High-volume workloads (classification, summarization, extraction) hit `gpt-4o-mini` rate limits quickly. Spreading across Azure OpenAI and two OpenAI accounts gives:

- **3× the tokens-per-minute limit**
- **Independent quotas** — Azure and OpenAI rate limit buckets don't share a counter
- **No code changes** — the app sends `model: "gpt-4o-mini"`; Portkey rewrites the destination

```json
{
  "name": "gpt-4o-mini-lb",
  "strategy": { "mode": "loadbalance" },
  "targets": [
    { "provider": "@azure-vk",    "override_params": { "model": "gpt-4o-mini" }, "weight": 1 },
    { "provider": "@openai-vk-1", "override_params": { "model": "gpt-4o-mini" }, "weight": 1 },
    { "provider": "@openai-vk-2", "override_params": { "model": "gpt-4o-mini" }, "weight": 1 }
  ]
}
```

### Target 4: `gemini-lb-with-fallback` — Load Balancer Inside a Fallback

The outer `fallback` has two entries:

1. **Primary**: an anonymous load balancer across three Gemini 2.5 Pro endpoints
2. **Fallback**: GPT-4.1 from OpenAI

When an individual Gemini target fails, the load balancer routes to one of the remaining two — no fallback triggered yet. Only when **all three Gemini endpoints fail** on the same request does the outer fallback activate and route to GPT-4.1.

```json
{
  "name": "gemini-lb-with-fallback",
  "strategy": { "mode": "fallback" },
  "targets": [
    {
      "strategy": { "mode": "loadbalance" },
      "targets": [
        { "provider": "@vertex-vk",   "override_params": { "model": "gemini-2.5-pro" }, "weight": 1 },
        { "provider": "@google-vk-1", "override_params": { "model": "gemini-2.5-pro" }, "weight": 1 },
        { "provider": "@google-vk-2", "override_params": { "model": "gemini-2.5-pro" }, "weight": 1 }
      ]
    },
    { "provider": "@openai-vk", "override_params": { "model": "gpt-4.1" } }
  ]
}
```

<Info>
The inner load balancer has no `name` field — anonymous nested targets are valid. Only top-level targets referenced by the conditional router require a `name`.
</Info>

## Using the Config

Pass the Config ID from [app.portkey.ai/configs](https://app.portkey.ai/configs) in requests:

<CodeGroup>

```python Python
from portkey_ai import Portkey

client = Portkey(
    api_key="PORTKEY_API_KEY",
    config="pc-multi-routing-xxxxx"
)

# Routes to claude-sonnet-lb (Anthropic + Vertex + Bedrock)
response = client.chat.completions.create(
    model="claude-sonnet",
    messages=[{"role": "user", "content": "Explain transformer architecture"}]
)

# Routes to gpt-4o-target (direct OpenAI)
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Write a unit test for this function"}]
)

# Routes to gpt-4o-mini-lb (Azure + 2× OpenAI)
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Classify this support ticket"}]
)

# Routes to gemini-lb-with-fallback (3× Gemini, fallback to gpt-4.1)
response = client.chat.completions.create(
    model="gemini-2.5-pro",
    messages=[{"role": "user", "content": "Analyze this 100k-token document"}]
)
```

## Virtual Keys

Create [Virtual Keys](https://app.portkey.ai/virtual-keys) for each provider and replace the slugs in the config:

| Slug | Provider | Notes |
|------|----------|-------|
| `@anthropic-vk` | Anthropic | Direct API |
| `@vertex-vk` | Google Vertex AI | Requires GCP credentials |
| `@bedrock-vk` | AWS Bedrock | Requires AWS credentials |
| `@openai-vk` | OpenAI | Primary key |
| `@openai-vk-1` | OpenAI | Second account |
| `@openai-vk-2` | OpenAI | Third account |
| `@azure-vk` | Azure OpenAI | Requires Azure deployment |
| `@google-vk-1` | Google AI Studio | First key |
| `@google-vk-2` | Google AI Studio | Second key |

<Info>
The `@slug` format references Virtual Keys in your Portkey workspace. Provider credentials can also be inlined using the `virtual_key` field directly in each target.
</Info>

## Observability

Every request is logged with its full routing path. In [Portkey Logs](https://app.portkey.ai/logs):

- Filter by **Config ID** to see all requests using this config
- Filter by **Trace ID** to see every attempt for a single request — including which load-balanced target was selected and whether a fallback triggered
- The **model** field reflects the actual provider model (not the alias), showing exactly which endpoint served the request

Add a `trace_id` for programmatic tracing:

```python
response = client.with_options(
    trace_id="user-req-20250514-abc123"
).chat.completions.create(
    model="gemini-2.5-pro",
    messages=[{"role": "user", "content": "Summarize this document"}]
)
```

## When to Use Each Pattern

| Pattern | Best for |
|---------|----------|
| **Conditional → single target** | Distinct models where one provider has sufficient capacity |
| **Conditional → load balancer** | High-volume models where a single provider's rate limits are a bottleneck |
| **Conditional → fallback(load balancer)** | Critical models requiring both throughput and zero-downtime guarantees |
| **Load balancer → fallback** | Distributing across primary providers with a cross-provider safety net |
