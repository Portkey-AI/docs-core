## What is a Guardrail?

LLMs can be unpredictable. You might receive a perfectly valid `200 OK` response from an API, but the content itself could break your app—perhaps it hallucinated, returned improperly formatted JSON, or included sensitive information. 

**Guardrails** are real-time protective checks that run on your LLM inputs (prompts) and outputs (responses). They ensure that the data flowing into and out of your LLMs strictly adheres to the rules you define. 

Built directly on top of the [Portkey Gateway](https://github.com/portkey-ai/gateway), Guardrails sit between your app and the LLM. When a check fails, you can automatically take action: deny the request, fall back to a different model, retry the prompt, or simply log the failure for later review.

Portkey offers over 20+ built-in guardrails, including:
* **JSON Schema Verification:** Ensure the response matches your exact JSON structure.
* **Regex Match:** Validate that text contains (or doesn't contain) specific patterns.
* **Gibberish Detection:** Prevent nonsensical outputs.
* **Prompt Injection Scans:** Protect your models from malicious user inputs.

## Quickstart

Putting Portkey Guardrails into production is a simple 4-step process. Here's the fastest way to get started:

<img src="/images/guardrails/flow.webp" />

### 1. Create a Guardrail Check

Head over to the **Guardrails** page in the Portkey UI and click **Create**. Select a check from the right sidebar (e.g., "JSON Schema" or "Regex Match"). Each check has a custom input field based on its use case.
*Note: A Guardrail must be configured to validate **EITHER** the Input (Prompt) **OR** the Output (Response), not both simultaneously.*

<Frame>
  <img src="/images/product/ai-gateway/ai-32.gif"/>
</Frame>

### 2. Define Guardrail Actions

Tell Portkey what to do when this check passes or fails. For a simple setup, you might just want to log the result asynchronously without blocking the request. Save your Guardrail to get a unique `$Guardrail_ID`.

<Frame>
  <img src="/images/product/ai-gateway/ai-33.png"/>
</Frame>

### 3. Add to a Portkey Config

To make your Guardrail active, attach it to a Portkey Config. You can apply them to inputs, outputs, or both. Portkey offers a simple way to configure this:

```json
{
  "input_guardrails": ["guardrails-id-123"],
  "output_guardrails": ["guardrails-id-456"]
}
```

### 4. Send Your Request

Finally, pass this Config ID when making a request using the Portkey SDK or API.

const portkey = new Portkey({

## How Guardrail Actions Work

When a Guardrail evaluates a request or response, it needs to know what to do next. This is governed by **Guardrail Actions**. Actions determine whether a check runs synchronously or asynchronously, what happens when it fails, and what status code Portkey should return.

### Core Behaviors

1. **Async execution (`async = TRUE`)**:
   The Guardrail check runs in the background. It adds zero latency to your request and does not interfere with the response. Portkey will simply log whether the check passed or failed. This is the **default state** and is perfect for monitoring LLM behavior without impacting users.

2. **Sync execution (`async = FALSE`)**:
   The Guardrail check runs *before* the request goes to the LLM (for inputs) or *before* the response returns to the user (for outputs). This adds slight latency but gives you full control to orchestrate the request based on the outcome.

### Impacting Outputs & Status Codes

When running synchronously, you must decide what happens if a check **FAILS**. This is controlled by the **Deny** setting.

| Guardrail Verdict | Deny Setting | Status Code Returned | Impact on Output |
| :--- | :--- | :--- | :--- |
| **PASS** | `TRUE` or `FALSE` | **200** | The request/response is processed normally and sent to the user. |
| **FAIL** | `FALSE` *(Default)* | **246** | The check failed, but the request/response is **STILL SENT** to the user. Portkey returns a `246` status code so your app knows a check failed without breaking the flow. |
| **FAIL** | `TRUE` | **446** | The check failed, and the request is **BLOCKED**. The user does not receive the LLM response. Portkey returns a `446` status code. |

#### Example Orchestration: Fallbacks & Retries

Using the `246` and `446` status codes, you can build powerful orchestration rules.

**Fallback to another model on failure:**
If your primary model fails a guardrail check (e.g., returns a 446), you can automatically fall back to a safer or smarter model.
```json
{
  "strategy": {
    "mode": "fallback",
    "on_status_codes": [246, 446]
  },
  "targets": [
    {"virtual_key": "openai-gpt-3.5"},
    {"virtual_key": "anthropic-claude-3"}
  ],
  "output_guardrails": ["guardrails-id-xxx"]
}
```

**Retry the request on failure:**
If a guardrail fails, simply retry the request again. This is particularly useful for JSON schema guardrails where the LLM might get it right on the next attempt.
```json
{
  "retry": {
    "on_status_codes": [246, 446],
    "attempts": 3
  },
  "output_guardrails": ["guardrails-id-xxx"]
}
```

### Feedback Actions

In addition to blocking or allowing requests, you can also append automated feedback based on the result:
* **On Success**: Automatically log your custom feedback to the request if all checks pass.
* **On Failure**: Automatically log your custom feedback to the request if any check fails.

This is highly recommended as it helps you build a robust "Evals dataset" over time, allowing you to measure your LLM's performance against your guardrails.
