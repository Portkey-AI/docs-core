### Information to Include

The more context you provide, the faster we can help. Include as much of the following as possible:

| Include | Why It Helps |
|---------|--------------|
| **Clear description** | What's happening vs. what you expected |
| **Steps to reproduce** | How can we recreate the issue? |
| **Error messages** | Exact error text or screenshots |
| **Trace IDs / Request IDs** | Enables quick lookup in our systems (redact sensitive data) |
| **Code snippets** | Relevant code causing the error |
| **Environment details** | SDK version, language, environment (dev/prod) |
| **Impact level** | How many users/requests are affected? |

### Tips for Faster Resolution

<Tip>
The more context you provide upfront, the faster we can help â€” fewer back-and-forth messages means quicker resolution.
</Tip>

- **Be specific** â€” Avoid vague descriptions like "it doesn't work"
- **Include trace IDs** â€” These enable instant lookup in our systems
- **One issue per thread** â€” Easier to track and resolve
- **Use threads** â€” Keep all discussion in the thread, not the main channel
- **Respond promptly** â€” Quick responses speed up resolution
- **Share files** â€” Screenshots, logs, and code samples help tremendously

## Understanding Ticket Statuses

Even though you're using Slack, your requests are tracked as tickets with statuses:

### Status Definitions

| Status | What It Means | What You Should Do |
|--------|---------------|-------------------|
| **On Portkey** | We're reviewing and working on your request | Wait for our response â€” we'll reply in the thread |
| **On You** | We've responded and need your input | Review our response and reply in the thread |
| **Closed** | Issue has been resolved | Reply in thread to reopen if needed |

### How Statuses Change

Statuses update **automatically** based on conversation activity:

- **When you start a thread** â†’ Status: "On Portkey"
- **When we reply asking for info** â†’ Status: "On You"
- **When you reply to our question** â†’ Status: "On Portkey"
- **When we resolve the issue** â†’ Status: "Closed"

<Note>
**No manual action needed.** The system automatically determines who needs to take the next action based on conversation context.
</Note>

### Reopening Closed Issues

If a closed issue resurfaces:

1. **Reply to the original thread** â€” This automatically reopens the ticket
2. **Or start a new thread** â€” Reference the original for context

## RBAC & Permissions

### Portal Access Levels

While Slack access is managed by your Slack workspace, the support portal has its own access levels:

| Permission | Admin | Member | No Access |
|------------|-------|--------|-----------|
| Login to portal | Yes | Yes | No |
| View own tickets | Yes | Yes | No |
| View all org tickets | Yes | No | No |

### Understanding Each Role

<AccordionGroup>
  <Accordion title="Admin" icon="user-shield" defaultOpen>
    **Full visibility and control**
    
    - View all tickets from all channels (including Slack)
    - See what teammates have submitted
    - Manage team access in the portal
    - Respond to any ticket on behalf of your organization
    
    **Ideal for:** Engineering leads, Support liaisons, IT administrators
  </Accordion>

  <Accordion title="Member" icon="user">
    **Personal ticket management**
    
    - View and manage only tickets you've personally submitted
    - Cannot see teammates' tickets in portal
    - Full ability to participate in Slack threads you're in
    
    **Ideal for:** Individual contributors, Developers
  </Accordion>

  <Accordion title="No Access" icon="user-slash">
    **No portal access**
    
    - Cannot log into the support portal
    - Can still participate in Slack channel if added
    - Slack threads create tickets visible to org Admins
    
    **Note:** Even without portal access, your Slack conversations are tracked.
  </Accordion>
</AccordionGroup>
